# Hanjify V2 轉換規則 — 設計草案

> 日期：2026-04-06
> 狀態：草案 — 僅供討論，尚未實作

---

## 1. 現行規則（V1）摘要

### 轉換流程

1. 將輸入文字依換行拆分，再依空格拆分為單詞
2. 對每個單詞：
   - 轉為小寫
   - 剝離前括號 `(` 及尾部標點（`,` `.` `;` `!` `?` `:` `)` 以及 `,"` `."` 等組合）
   - 在 `wordList` 字典中查找（約 4,793 筆）
   - 若查無，依序嘗試剝離後綴：`s`、`ing`、`ed`、`es`（命中即停）
   - 特殊情況：大寫 `"I"` 始終保留
3. 查到的詞 → `<ruby>漢字<rt>english</rt></ruby>`
4. 查不到的詞 → 原文顯示

### 優點

- **優雅降級**：未知詞直接顯示原文英語，不會破壞輸出
- **零依賴**：單一 HTML 檔案，無建置步驟，無框架
- **Ruby 標注**：運用 CJK 排版慣例的振假名標記，可切換顯示
- **混合來源漢字**：涵蓋簡體中文、繁體中文、日文漢字、韓文漢字及古字——呈現「自然演化」的質感，不侷限於單一文化

### 弱點

- **僅 4 條後綴規則**：許多常見詞形查不到（如 `running`、`cities`、`happier`）
- **無拼寫還原**：雙寫子音（`running` → `runn`）、y→ies 變化均未處理
- **無前綴處理**：`unhappy`、`redo`、`overwork` 等無法拆解
- **無複合詞拆分**：`bookshelf`、`sunshine` 作為整詞查找會失敗
- **硬編碼標點清單**：遺漏智慧引號、長破折號、省略號等
- **字典不一致**：同一語素在相關詞彙中可能使用不同漢字

## 2. V2 設計提案

### 2.1 分層詞形分析（Layered Morphological Analysis）

改進空間最大的環節。以分層系統取代現有的 4 條後綴規則：

#### 第一層：前綴剝離（可選）

嘗試移除常見前綴，查找剩餘部分：

```
un-    re-    pre-    dis-    mis-
over-  under- out-    non-
```

若前綴與剩餘部分皆可解析，顯示為：`un幸福`（前綴作為送假名置於漢字前方）。

#### 第二層：詞根查找（字典）

與 V1 相同——直接在字典中查找基本形。

#### 第三層：後綴剝離（含拼寫還原）

依優先順序嘗試後綴，每條後綴附帶還原規則：

| 後綴 | 還原方式（依序嘗試） |
|------|---------------------|
| `-ies` | → 替換為 `-y`（cities → city） |
| `-ied` | → 替換為 `-y`（carried → carry） |
| `-ing` | → (1) 直接移除 (2) 移除 + 去雙寫子音（running → run）(3) 移除 + 補 `e`（making → make） |
| `-ed`  | → (1) 直接移除 (2) 去雙寫（stopped → stop）(3) 補 `e`（loved → love） |
| `-er`  | → (1) 直接移除 (2) 去雙寫（bigger → big）(3) 補 `e`（nicer → nice） |
| `-est` | → 同 `-er` 邏輯 |
| `-es`  | → 直接移除 |
| `-s`   | → 直接移除 |
| `-ly`  | → 直接移除 |
| `-ful` | → 直接移除 |
| `-less`| → 直接移除 |
| `-ness`| → 直接移除 |
| `-ment`| → 直接移除 |
| `-able`/`-ible` | → 直接移除；另嘗試補 `e`（lovable → love） |
| `-tion`/`-sion` | → 嘗試還原動詞形（creation → create） |
| `-ous` | → 直接移除 |

**去雙寫規則**：若詞幹以雙寫子音結尾（如 `runn`），移除最後一個字母，檢查結果是否在字典中。

**設計限制**：每個後綴最多嘗試 3 種還原變體。若皆不匹配，移至下一個後綴。若所有後綴皆失敗，回退為原文顯示。

### 2.2 送假名系統（正式化）

V1 已隱含送假名概念——被剝離的後綴會出現在漢字後方。V2 將此概念明確化：

- **漢字** ＝ 語義核心（來自字典）
- **送假名** ＝ 文法/派生後綴（英文文字）

顯示範例：

| 輸入 | 詞根 | 送假名 | 顯示 |
|------|------|--------|------|
| running | run＝走 | -ing | 走ing |
| carefully | care＝注意 | -fully | 注意fully |
| unhappy | happy＝幸福 | un- | un幸福 |
| cities | city＝都市 | -ies | 都市ies |
| unbelievable | believe＝信 | un- + -able | un信able |

同時具有前綴與後綴時，兩者皆作為送假名顯示在漢字核心的兩側。

### 2.3 複合詞拆分

當單詞不在字典中時，嘗試拆分為兩個已知詞：

```
演算法：
  for i in range(len(word)-1, 2, -1):    # 貪心法，從最長前綴開始
    prefix = word[:i]
    suffix = word[i:]
    if prefix in wordList and suffix in wordList:
      return compose(wordList[prefix], wordList[suffix])
```

範例：
- `bookshelf` → `book`（冊）+ `shelf`（棚）→ 冊棚
- `sunshine` → `sun`（日）+ `shine`（輝）→ 日輝
- `doorbell` → `door`（門）+ `bell`（鈴）→ 門鈴

**限制**：僅拆分為恰好 2 部分。三段式拆分容易產生誤判。

### 2.4 標點處理

以正規表達式取代硬編碼清單：

```javascript
const LEADING_PUNCT = /^[("''"\u00BF\u00A1]+/;
const TRAILING_PUNCT = /[,.:;!?)"''"—\u2026\-]+$/;
```

涵蓋智慧引號、長破折號、省略號及倒置標點符號。

### 2.5 大小寫策略

| 模式 | 行為 |
|------|------|
| 全大寫（如 `NASA`、`FBI`） | 跳過轉換——視為專有名詞/縮寫 |
| 首字母大寫（如 `The`、`Happy`） | 正常轉換，ruby 中保留原始大小寫 |
| 獨立的 `"I"` | 特殊處理——保持原樣（同 V1） |

### 2.6 選字哲學

延續 V1 的核心哲學：

> 「如果一個英語母語者懂一些漢字，翻字典來轉寫英文，會寫出什麼？」

新增一條原則：

> **詞根一致性**：同一語素在所有派生形式中應使用相同的漢字。

V1 中的問題範例：
- `act` ＝ 演、`action` ＝ 行動、`active` ＝ 活動的 —— 三個詞用了三套不同的漢字基底

V2 原則：若 `act` ＝ 演，則 `action` 應從「演」派生（如「演動」），而非引入完全不同的中文翻譯。

這主要是**字典維護**層面的問題，而非程式碼變更。但改良後的詞形分析能減少手動添加每個派生形式的需求，間接支持此原則。

## 3. V1 與 V2 對照

| 面向 | V1 | V2 |
|------|----|----|
| **後綴剝離** | 4 種後綴，無拼寫還原 | 15 種以上後綴，含去雙寫及補 e 還原 |
| **前綴處理** | 無 | 9 種常見前綴 |
| **送假名** | 隱含，僅限被剝離的後綴 | 明確系統，涵蓋前綴與後綴 |
| **複合詞** | 不處理 | 貪心二段式拆分 |
| **標點** | 硬編碼 13 種模式 | 正規表達式，可擴展 |
| **大小寫** | 僅處理 `"I"` | 全大寫跳過 ＋ 首字母大寫保留於 ruby |
| **字典結構** | 扁平鍵值對 | 結構相同，但加入詞根一致性原則 |
| **降級機制** | 原文顯示 | **相同** |
| **架構** | 單一 HTML 檔案，無建置 | **相同** |
| **Ruby 顯示** | `<ruby>` + `<rt>` | **相同** |
| **漢字來源** | 混合 CJK 來源 | **相同哲學** |

## 4. V2 不會改變的部分

以下 V1 的設計決策是正確的，應予以保留：

1. **無建置步驟** —— 純 HTML/CSS/JS
2. **單檔架構** —— 所有邏輯集中在 `index.html` + `wordlist.js`
3. **優雅降級** —— 未知詞原文顯示
4. **Ruby 標注** —— 標準 CJK 排版慣例
5. **混合來源漢字** —— 不侷限於任何單一 CJK 傳統
6. **「不是翻譯」** —— 漢字化保留英語文法，僅替換詞彙
7. **趣味優先** —— 目標是帶來樂趣與猜謎體驗，而非語言學上的完美

1. **高影響、中等工程量**：擴展後綴剝離與拼寫還原

3. **中等影響、中等工程量**：複合詞拆分
4. **中等影響、高工程量**：前綴處理（需調整送假名顯示邏輯）
5. **持續進行**：字典詞根一致性稽核（維護工作，非程式碼）
